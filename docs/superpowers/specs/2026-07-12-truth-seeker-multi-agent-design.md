# truth-seeker 多 Agent 架构设计

**日期**: 2026-07-12
**版本**: v2.0.0

---

## 1. 动机

当前 truth-seeker 是单 agent 技能，所有工作（搜索、审计、推理、报告生成）在一次调用中串行完成。问题：

- **上下文过载**：搜索大量网页 + 阅读详情 + 推理 + 生成 HTML，上下文窗口压力大
- **职责混杂**：数据收集和推理报告混在一起，难以独立优化
- **产出冗余**：终端文本报告和 HTML 报告内容重复，浪费 token

## 2. 架构概览

将 truth-seeker 拆分为 **编排者 + 两个专职 Agent** 的分阶段流水线：

```
用户输入
    │
    ▼
┌─────────────────────────────────────────┐
│           SKILL.md（编排者）              │
│                                         │
│  1. 接收用户输入，不做预处理               │
│  2. 调用 Agent 1 → 研究员                │
│  3. 传递结构化数据给 Agent 2              │
│  4. 调用 Agent 2 → 报告生成               │
│  5. 保存 HTML → 自动打开浏览器             │
└─────────────┬───────────────┬───────────┘
              │               │
              ▼               ▼
     ┌──────────────┐  ┌──────────────┐
     │ Agent 1 研究员 │  │ Agent 2 报告  │
     │              │  │              │
     │ 拆解陈述      │  │ 贝叶斯推理    │
     │ 多角度搜索    │  │ 竞争假设      │
     │ 五维来源审计  │  │ 原理分析      │
     │ 独立性检查    │  │ 底层逻辑      │
     │ 搜索质量自检  │  │ 生成 HTML     │
     │              │  │              │
     │ 输出: JSON   │  │ 输入: JSON   │
     └──────────────┘  └──────────────┘
```

## 3. 数据契约

Agent 1 产出结构化 JSON，作为 Agent 2 的唯一输入。

```json
{
  "original_claim": "用户原始输入",
  "decomposition": {
    "claims": [
      {
        "id": 1,
        "text": "主张内容",
        "type": "factual | opinion | mixed",
        "falsifiable": true,
        "search_keywords": {
          "zh": ["关键词1"],
          "en": ["keyword1"]
        }
      }
    ],
    "summary": "拆解小结"
  },
  "evidence": [
    {
      "claim_id": 1,
      "supporting": [
        {
          "description": "证据描述",
          "source_name": "来源名称",
          "source_type": "学术期刊 | 政府报告 | 媒体报道 | 专业机构 | 网络文章 | 社交媒体",
          "url": "https://...",
          "stars": 4,
          "audit": {
            "methodology": "high | medium | low",
            "timeliness": "high | medium | low",
            "independence": "high | medium | low",
            "conflict_of_interest": "有 | 疑似 | 无"
          },
          "summary": "一句话摘要"
        }
      ],
      "contrary": []
    }
  ],
  "media_materials": [
    {
      "type": "image | pdf | dataset",
      "title": "材料标题",
      "url": "https://...",
      "source": "来源"
    }
  ],
  "search_quality": {
    "language_coverage": "全面 | 部分 | 单一",
    "region_coverage": "全面 | 部分 | 单一",
    "tool_limitation": "无影响 | 部分影响 | 显著影响",
    "note": "搜索质量自检说明"
  },
  "researcher_note": "研究员对搜索过程和发现的自然语言总结"
}
```

### 数据契约设计原则

- **结构化字段**（claim、evidence、audit、search_quality）确保 Agent 2 可以精确读取，不依赖自然语言解析
- **researcher_note** 保留自然语言灵活性，传递搜索过程中的洞察和注意事项
- **不包含推理结论**：Agent 1 只交付原始数据，真伪判断完全由 Agent 2 完成

## 4. Agent 1：研究员

### 职责边界

**只收集和整理，不推理、不判断、不生成报告。**

### 输入

用户原始陈述（字符串），无额外上下文要求。

### 工作流程

```
1. 拆解陈述
   ├─ 识别核心主张（一个陈述可能包含多个主张）
   ├─ 区分 fact / opinion / mixed
   ├─ 预判可证伪性
   └─ 提炼中英文两组关键词

2. 执行搜索（每个主张 × 6 种搜索角度）
   ├─ 正面搜索（支持该主张的证据）
   ├─ 反面搜索（质疑/反驳该主张的证据）
   ├─ 溯源搜索（该主张的原始出处和传播链）
   ├─ 权威源定向搜索
   ├─ 媒体材料搜索（图表、PDF、官方截图）
   └─ 中英文各至少 2 组不同关键词组合

3. 获取详情
   └─ 对搜索结果中的关键页面使用 WebFetch 获取完整内容

4. 五维来源审计
   └─ 每条证据标注：来源类型 / 方法论质量 / 时效性 / 独立性 / 利益冲突
      └─ 重点检查"同源多引"——5 篇报道引用同一份原始报告 = 1 个来源

5. 搜索质量自检
   └─ 评估语言覆盖度 / 地域覆盖度 / 工具限制影响
```

### 关键约束

- 不跳过搜索：每个 claim 至少 6 种角度搜索
- 双语强制：中文和英文关键词各至少 2 组，不能只搜一种语言
- 不含推理：`stars` 字段是基于来源类型的客观评级，不是对证据力度的主观判断
- 独立性标注：如果多条证据来自同一原始来源，在 `audit.independence` 中标注

### 使用的工具

`WebSearch`、`WebFetch`

## 5. Agent 2：报告生成

### 职责边界

**只推理和生成，不做新搜索（除非发现明显证据缺口）。**

### 输入

Agent 1 产出的结构化 JSON。

### 工作流程

```
1. 贝叶斯推理（对每个 claim）
   ├─ 设定先验概率（仅基于常识和基础率，Agent 1 的证据尚未引入）
   ├─ 评估每条证据的似然比
   │   ├─ 强支持 (±30%)：权威 RCT / 官方统计 / 多源一致
   │   ├─ 中等支持 (±10%)：调查报道 / 专业机构报告
   │   ├─ 弱支持 (±2%)：个案报道 / 专家意见
   │   ├─ 弱反对 (−5%)：缺乏证据
   │   └─ 强反对 (−30%)：权威研究否定 / 官方辟谣
   ├─ 计算后验可信度（百分比）
   └─ 给出 95% 置信区间
      └─ 区间宽度由 Agent 1 的 search_quality 决定
         └─ 覆盖面越窄，区间越宽

2. 竞争假设与效应量
   ├─ 列出替代解释（混淆变量 / 反向因果 / 选择偏倚）
   ├─ 评估每种替代解释的可能性（likely / possible / unlikely）
   ├─ 估算效应量（大 / 中 / 小 / 可忽略 / 反方向）
   └─ 用具体数字量化影响幅度

3. 原理与底层逻辑
   ├─ 因果机制分析：解释"为什么是这样"，不是堆砌百科知识
   ├─ 逻辑谬误识别：因果混淆 / 以偏概全 / 幸存者偏差 等
   ├─ 数据解读：统计口径 / 对比基准 / 是否被断章取义
   ├─ 传播溯源：这个说法为什么会流行
   └─ 利益相关方分析：谁在推动这个说法

4. 生成 HTML 报告
   ├─ 读取 references/blocks/index.md 了解装配顺序
   ├─ 按需读取所需 section 块
   ├─ 内联 styles.css
   └─ 组装为自包含的单文件 HTML
```

### 关键约束

- 不做新搜索：依赖 Agent 1 的数据，除非发现明显证据缺口才补充搜索
- 推理透明：每条结论说清楚"先验 → 证据 → 后验"链条
- 覆盖度传导：`search_quality` 直接影响置信区间宽度
- 不确定就说不确定：证据不足时标注"无法判断"，不强行结论

### 输出

- 生成自包含的单文件 HTML 报告
- **不再输出终端纯文本报告**（省 token）

### 使用的工具

`Read`（读取 blocks/ 下的模板文件）、`Write`（保存 HTML 文件）、`Bash`（打开浏览器）

## 6. 编排者（SKILL.md）

### 职责

轻量编排，不做业务逻辑。

### 流程

```
1. 接收用户输入
2. 使用 Agent 工具调用 Agent 1（研究员），传入用户输入
3. 解析 Agent 1 返回的结构化 JSON
4. 使用 Agent 工具调用 Agent 2（报告生成），传入 Agent 1 的 JSON
5. 将 Agent 2 返回的 HTML 内容写入文件
6. 自动在浏览器中打开 HTML 文件
```

### 关键约束

- 编排者本身不做搜索、不做推理、不读模板
- 编排者仅负责传递数据和调度

## 7. 文件变更范围

| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `SKILL.md` | 重写 | 改为编排者角色，定义两个 Agent 的调用流程和数据契约 |
| `references/blocks/` | 不变 | HTML 模板块由 Agent 2 读取，结构不变 |
| 新增 Agent 定义文件 | 新增 | 可能需要 `.claude/agents/` 下定义 Agent 1 和 Agent 2 的配置 |

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Agent 1 返回的 JSON 格式不完整 | 数据契约定义必填字段，Agent 2 入参前校验 |
| Agent 2 发现证据缺口需要补充搜索 | Agent 2 可以做补充搜索（作为例外，不是常态） |
| 两个 Agent 调用增加总耗时 | Agent 1 搜索是耗时主要来源，Agent 2 纯推理快，总耗时变化不大 |
| JSON 数据量大，传递时截断 | Agent 1 对每条证据只保留摘要而非全文，控制 JSON 大小 |
